- JPA란?

  : JPA(Java Persistence API)란, **자바 진영에서 ORM(Object-Relational Mapping) 기술의 표준으로 사용되는 인터페이스의 모음 (워크북에 나와있는 설명)**

  ***ORM: 객체와 DB 테이블을 자동으로 매핑**해주는 기술.
  ⇒ SQL을 직접 작성하지 않아도 DB를 다룰 수 있음.

  자바 데이터베이스를 다룰 때 SQL 문을 직접 작성하는 **JDBC** 방식을 주로 사용했지만, 이는 반복적인 코드가 많고 객체지향적인 설계를 데이터베이스에 반영하기 어렵다는 단점이 존재.

  **JPA의 특징과 장점**

    - **생산성 향상**: SQL문을 일일이 짤 필요 없이 간단한 코드 한 두줄로 해결 가능.
    - **유지 보수**: 필드가 변경되었을 때, SQL을 수정할 필요 없음.
      ⇒ 클래스만 수정하면 JPA가 알아서 SQL생성.
    - **데이터 독립성**: 특정 DB에 종속되지 않아 DB 교체에 용이.
    - **패러다임 불일치 해결**: 객체 지향의 특징(상속, 연관관계 등등..)을 DB테이블 구조에 맞게 잘 풀어냄.

  **Hibernate:** JPA라는 명세를 실제로 코드로 구현한 가장 대표적인 프레임워크

- N+1 문제란?

  : 1번 조회해야 할 것을 N+1번 조회하는 것.
  ⇒ 성능 문제!

  **JPA와 같은 ORM 기술**을 사용할 때 가장 흔하게 발생하는 **성능 저하의 요인.**

  **문제 발생 시나리오**

  '게시글(Post)'과 '댓글(Comment)'이 1:N 관계인 상황.

    1. 사용자가 모든 게시글을 조회하려고 `findAll()`을 호출.
    2. JPA는 우선 `SELECT * FROM Post`라는 **쿼리 1개**를 날려 게시글 목록을 가져옴.
    3. 만약 게시글이 10개(N개)라면, 각 게시글에 달린 댓글을 조회하기 위해 **추가로 10번의 쿼리**가 더 실행!
        - *1번 게시글의 댓글 조회... (Query 1)*
        - *2번 게시글의 댓글 조회... (Query 2)*
        - ...
        - *10번 게시글의 댓글 조회... (Query 10)*
    4. 결과적으로 쿼리를 1번이 아닌 11번(N+1) 날림.

  **발생 원인**

  JPA의 기본 전략 중 하나인 **지연 로딩 때문.**
  ⇒ 하지만, 즉시 로딩(Eager Loading)을 쓴다고 해결되지는 않음.

  **해결 방법**

    1. **Fetch Join:**

    ```java
    public interface TeamRepository extends JpaRepository<Team, Long> {
    		@Query("SELECT t FROM Team t JOIN FETCH t.members")
    		List<Team> findAllWithMembers();
    }
    ```

  ⇒ JPQL에서 `join fetch` 키워드를 사용하여 연관된 엔티티를 한 번의 `INNER JOIN`으로 묶어서 가져옴.

    1. **@EntityGraph:**

  어노테이션을 통해 특정 쿼리 실행 시 연관된 엔티티를 함께 조회하도록 설정.

- 지연로딩과 즉시로딩의 차이는?

  **연관된 엔티티를 언제 조회할지 시점의 차이.**

  ### 1. 즉시 로딩 (Eager Loading)

  데이터를 조회할 때, **연관된 객체까지 한꺼번에** 다 조회.

  `@ManyToOne(fetch = FetchType.EAGER)`

  ⇒ `Member`를 조회하면, JPA가 내부적으로 `JOIN` 쿼리를 사용해 연관된 `Team` 데이터까지 즉시 가져옴.

  **특징**

    - 객체를 가져온 시점에 이미 연관 데이터가 메모리에 다 올라와 있음.
    - 테이블이 복잡하게 얽혀 있을 때 사용하면, 예상치 못한 거대한 JOIN 쿼리가 발생하거나 **N+1 문제**가 발생할 확률이 높음.

  ### 2. 지연 로딩 (Lazy Loading)

  연관된 객체는 일단 **가짜(Proxy) 객체**로 채워두고, **실제로 그 데이터가 필요한 시점**에 쿼리를 날려 조회하는 방식

  `@ManyToOne(fetch = FetchType.LAZY)`

  ⇒ `Member`를 조회하면 일단 `Member` 데이터만 가져옴. 이후 코드에서 `member.getTeam().getName()` 처럼 팀의 구체적인 필드를 건드리는 순간 DB에 쿼리를 날림.

  **특징**

    - 불필요한 데이터 조회를 방지하여 성능을 최적화.
    - 연관 객체를 사용하는 시점에 DB 연결(세션/트랜잭션)이 유지되어 있어야 함.

  *실무에서는 모든 연관관계를 LAZY로 설정.
  이후, 특정 화면에서 연관된 데이터가 한꺼번에 필요하다면, **Fetch Join**이나 **Entity Graph**를 사용하여 그 시점에만 일시적으로 묶어서 가져옴.

- JPQL란?

  : JPA에서 사용하는 **객체지향 쿼리 언어.**

  일반적인 SQL이 **데이터베이스 테이블**을 대상으로 쿼리를 날린다면, JPQL은 **엔티티 객체**를 대상으로 쿼리를 날림.

  **SQL vs JPQL 차이점**

  | **구분** | **SQL** | **JPQL** |
      | --- | --- | --- |
  | **조회 대상** | **데이터베이스 테이블** | **엔티티 객체** |
  | **문법 특징** | 대소문자 구분 없음 (DB마다 다름) | 엔티티와 필드는 **대소문자 엄격 구분** |
  | **별칭 (Alias)** | 선택 사항 | **필수** (예: `Member AS m`) |

  ### JPQL의 주요 특징

    - **방언(Dialect) 독립성:** JPQL은 특정 DB의 SQL 문법에 종속되지 않음. JPQL을 작성하면 JPA가 설정된 DB 방언에 맞춰 Oracle, MySQL, PostgreSQL 등 각각의 SQL로 자동 변환.
    - **객체지향적:** 상속 관계에 있는 엔티티를 조회하거나, 연관된 객체를 타고 들어가는 그래프 탐색이 가능.
    - **강력한 추상화:** 복잡한 조인이나 집계 함수(`SUM`, `AVG`, `GROUP BY` 등)도 표준 SQL과 거의 유사한 문법으로 지원.

- Fetch Join란?

  : **연관된 엔티티를 한 번의 쿼리로 함께 조회**하는 JPQL 문법.

  ⇒ **N+1 문제를 해결하기 위한 가장 강력하고 대표적인 방법.**

  ### 장점

    - **성능 최적화:** 여러 번 나갈 쿼리를 단 한 번의 조인 쿼리로 끝냄.
    - **객체 그래프 유지:** 연관된 객체들을 한 번에 메모리에 올려서 객체 간의 탐색이 자유로움.

  ### 단점

    - **페이징(Paging) API 사용 불가 (컬렉션 페치 조인 시):**
      1:N 관계(예: 게시글과 댓글)에서 페치 조인을 쓰고 `setFirstResult`, `setMaxResults` 같은 페이징 처리를 하면, 하이버네이트는 모든 데이터를 메모리에 끌어올린 뒤 메모리에서 페이징을 시도. 이는 데이터가 많을 경우 치명적인 메모리 장애(OutOfMemory)를 유발.
    - **둘 이상의 컬렉션은 페치 조인 금지:** 1:N:M 처럼 여러 컬렉션을 동시에 페치 조인하면 데이터가 기하급수적으로 뻥튀기(카테시안 곱)되어 예상치 못한 결과가 나올 수 있음.
    - **별칭(Alias) 사용 자제:** JPA 표준 명세상 페치 조인 대상에는 별칭을 줄 수 없으며, 가급적 쿼리 조건절(`WHERE`)에서 사용하지 않는 것이 원칙.

- @EntityGraph란?

  : **Fetch Join을 어노테이션으로 간편하게** 사용하는 방법.

    ```java
    // Fetch Join - JPQL 직접 작성
    @Query("SELECT m FROM Member m JOIN FETCH m.team")
    List<Member> findAllWithTeam();
    
    // @EntityGraph - 어노테이션으로 간편하게
    @EntityGraph(attributePaths = "team")
    @Query("SELECT m FROM Member m")
    List<Member> findAllWithTeam();
    
    // @EntityGraph - @Query도 생략 가능
    @EntityGraph(attributePaths = "team")
    List<Member> findAll();
    ```

  ⇒ 결과는 동일.

  **비교**

  @EntityGraph

    - 단순한 연관관계 조회
    - JPQL 작성하기 귀찮을 때

  Fetch Join

    - 복잡한 조건이 필요할 때
    - 컬렉션 조회 + 페이징

  ### 주의사항

    - **Outer Join의 특성:** `@EntityGraph`는 기본적으로 `LEFT OUTER JOIN`을 사용. 만약 교집합 데이터만 필요한 `INNER JOIN` 성능 최적화가 필수적이라면 직접 JPQL에 `FETCH JOIN`을 쓰는 것이 좋을 수 있음.

- commit과 flush 차이점은?

  ### 1. Flush (플러시)

  플러시는 영속성 컨텍스트(JPA의 임시 저장소)의 변경 내용을 **데이터베이스에 전달**하는 과정.

  (*영속성 컨텍스트: JPA가 관리하는 임시 공간)

    - **하는 일:** 쓰기 지연 SQL 저장소에 쌓여있는 `INSERT`, `UPDATE`, `DELETE` 쿼리를 DB에 날림.
    - **특징:**
        - 플러시가 일어난다고 해서 데이터가 DB에 완전히 저장되는 것은 아님. (임시 저장 상태)
        - **영속성 컨텍스트를 비우지 않음.** 컨텍스트 내의 엔티티들은 그대로 유지.
        - 데이터베이스 트랜잭션이 종료되지 않았으므로, 문제가 생기면 여전히 롤백이 가능.
    - **발생 시점:**
        1. 직접 호출 (`em.flush()`)
        2. 트랜잭션 커밋 시 (자동 호출)
        3. JPQL 쿼리 실행 시 (자동 호출)

  ### 2. Commit (커밋)

  커밋은 데이터베이스의 트랜잭션을 끝내고 변경 내용을 **물리적인 DB에 영구적으로 반영**하는 것.

    - **하는 일:** DB 트랜잭션을 종료.
    - **특징:**
        - 커밋이 실행되면 더 이상 작업을 취소(Rollback)할 수 없음.
        - JPA에서는 커밋이 호출될 때 **자동으로 플러시를 먼저 수행**하여 변경 내용을 DB에 전달한 뒤 커밋을 진행.

  **workflow**

  Entity 변경
  ↓
  영속성 컨텍스트
  ↓ flush
  DB에 SQL 전송
  ↓ commit
  DB에 영구 저장
